## Goal

Ship the booking machinery end-to-end. From a saved trip, the user clicks "Book this â†’" â†’ BookingAgent produces a structured `BookingDraft` (dates, guest count, total, cancellation policy) â†’ an approval modal shows the draft â†’ on confirm, `BookingProvider.book(draft)` creates a `Booking` and the user lands on `/bookings/[bookingId]` with the confirmation. Mock-safe end-to-end via `MockBookingProvider`. Real provider booking + autonomous mode + card capture are explicitly **deferred to D.x** - bookings produce real, irreversible state changes; the architectural seam lands here, the live wiring + premium gate land separately.

## What shipped

### Core booking types (`src/core/booking.ts`)
- `BookingStatus = 'draft' | 'confirmed' | 'canceled' | 'failed'`.
- `CancellationPolicy` discriminated union: `free-until`, `partial-refund`, `non-refundable`. Each carries the structured fields we evaluate `isCancelable()` against, plus a verbatim provider description string.
- `TravelerInfo` (primaryName, email, guestCount.{adults,children,infants}). Email validated against RFC.
- `BookingDraft` - pre-confirm shape with `idempotencyKey` (the user's confirm-click token), dates, total, cancellation policy, traveler, owner, savedTripId, stayId, providerId, `placeholderDates: boolean`.
- `Booking` - post-provider shape with `providerBookingRef`, `status`, `confirmedAt`, `canceledAt`, `failureReason`. Mirrors the durable record an admin queries.
- `OwnerKey` re-defined here (not imported from auth) to keep core layer-clean - same approach as C4's billing.
- `BookingError` - typed reasons: `unknown-draft`, `unknown-booking`, `already-confirmed`, `not-cancelable`, `not-owner`, `idempotency-collision`, `provider-error`, `invalid-traveler`. Routes map these to HTTP statuses without inspecting message strings.
- `mintIdempotencyKey()`, `mintBookingId()`, `mintBookingDraftId()` - stable random ids with prefixes (`bk_`, `bok_`, `bkd_`).
- `isCancelable(booking, now)` - derived from status + policy + clock.

### `BookingProvider` interface + impls (`src/lib/booking/`)

**Interface** (`booking-provider.ts`):
```ts
interface BookingProvider {
  kind: 'mock' | 'booking-com' | 'expedia';
  book(draft: BookingDraft): Promise<Booking>;       // idempotent on draft.idempotencyKey
  cancel({ bookingId, reason? }): Promise<Booking>;  // idempotent
  getBooking(bookingId: string): Promise<Booking | null>;
}
```

**`MockBookingProvider`** (`mock-booking-provider.ts`):
- Internal `Map<idempotencyKey, Booking>` so `book` is genuinely idempotent - same key returns the same `Booking` across calls.
- `cancel` flips status to `canceled` + stamps `canceledAt`. Second cancel is a no-op (returns the canceled booking unchanged).
- `STAYSCOUT_BOOKING_FAIL=1` makes the next `book` call throw `BookingError('provider-error')`, then self-clears so the retry succeeds. Useful for manual testing of the failure path.

### Stores (`src/lib/booking/`)
- `BookingStore` interface - `putDraft` / `getDraft` (by idempotencyKey), `putBooking` / `getBooking` (owner-gated by id), `listByOwner`, `listAll` (admin).
- `InMemoryBookingStore` - process-singleton via globalThis. Three indexes: drafts by key, bookings by id, bookings per-owner ring (push-order, reversed for most-recent-first listing). Owner-mismatch returns null from `getBooking` (defense in depth). Updating a booking in place keeps the per-owner bucket de-duplicated.

### `BookingAgent` (`src/agents/booking-agent.ts`)
- Three operations: `draftBooking(input)`, `confirmBooking({ ownerKey, idempotencyKey })`, `cancelBooking({ ownerKey, bookingId, reason? })`.
- `draftBooking`:
  - Resolves dates from `trip.intent.dates`. Specific dates â†’ use as-is. Anything else â†’ today + 30 days for checkIn, +nights for checkOut, with `placeholderDates: true` so the modal flags it.
  - Computes total from `pricing.totalForStay` if present, else `pricePerNight Ã— nights`.
  - Applies a `free-until = checkIn âˆ’ 7 days` default cancellation policy. Real providers will return their own in D.x.
  - Mints `idempotencyKey`, persists via `store.putDraft`. **No provider call.**
- `confirmBooking`:
  - Pulls draft, verifies ownerKey matches â†’ `BookingError('not-owner')` otherwise.
  - Calls provider's idempotent `book(draft)`. On error: persists a `failed` Booking row with `failureReason` set so admins can see what was attempted, then re-throws.
  - Persists the resulting Booking.
- `cancelBooking`:
  - Owner-gated read. Idempotent at the agent level (returns the canceled booking unchanged on a second call).
  - Validates `isCancelable(booking, now)`. Calls provider's idempotent `cancel`. Preserves our canonical id (provider returns its own ref-keyed booking).
- Slice D always passes `autonomous: false` from the API. The flag exists for D.x to flip behind a premium gate.

### API routes (`src/app/api/bookings/`)
- `POST /api/bookings/draft` - owner-gated (401 for anonymous), body `{ savedTripId, traveler }` Zod-parsed, 404 if trip not owned, returns `{ draft }`.
- `POST /api/bookings/confirm` - owner-gated, body `{ idempotencyKey }`, status mapping: 404 unknown-draft, 403 not-owner, 502 provider-error, 400 other.
- `POST /api/bookings/[bookingId]/cancel` - owner-gated, optional body `{ reason }`, status mapping: 404 unknown-booking, 409 not-cancelable.

All three require sign-in (no anonymous bookings - Slice D needs a stable owner key for the confirmation page + admin trail).

## Architectural decisions (recorded)

**1. Approval-gated, every booking, every tier.** Slice D ships *only* the approval-gated path. The `autonomous` flag on the agent's input exists; the API never passes it. Autonomous mode is a separate policy decision - it gets its own design pass + premium gate via the existing `requirePremium(owner)` from C4 + a deliberate "you've authorized autonomous" affordance in D.x.

**2. Idempotency on the user's confirm click.** The first thing `draftBooking` does is mint an `idempotencyKey` and stamp it on the draft. The provider's `book` is keyed on it; the store dedupes by it. Double-clicks coalesce to a single booking - same discipline as C4's webhook idempotency.

**3. Provider state is the source of truth (in spirit).** Slice D's mock provider holds bookings in-process; we cache the same shape in `BookingStore` so reads are fast. D.x's real providers are remote - `BookingStore` becomes a cache, `provider.getBooking(id)` is the canonical read on stale-suspect rows.

**4. No card capture in-app.** Slice D records `total.amount` as what *would* be charged. Mock provider just confirms. Real-mode integrations (D.x) hand off to provider Checkout for card capture. PCI scope = zero, matching C4.

**5. Mock-safe + opt-in real.** `MockBookingProvider` is the default; every authed user can complete the full flow without keys. `STAYSCOUT_LIVE_BOOKING=1` is read + surfaced (`liveEnabled`) but doesn't activate live providers in Slice D - the seam is here, the wiring isn't.

**6. Bookings tie to owners, not to saved trips.** Cascade rules: deleting a saved trip leaves bookings intact (you don't un-book by un-saving). The confirmation page falls back to placeholder strings when the trip is gone.

**7. Cancellation is part of the contract.** `BookingProvider.cancel(id, reason)` is required. Mock provider's mock cancellation is "fully refundable in dev." Real providers return refund eligibility per their policy.

**8. Admin-visible from day one.** `/admin/bookings` ships in Slice D alongside the user-facing flow. Operators have eyes on a side-effecting feature from the moment it ships - not added later.

**9. No new OrchestratorEvent kinds.** Bookings are out-of-band from the turn flow, same as itineraries (C3) and monitoring (C2). The orchestrator engine, intent agent, and providers are untouched.

**10. Anonymous can't book.** `/api/bookings/draft` returns 401 for `ownerKind === 'session'`. Sign-in is required - Slice D needs a stable owner key for the confirmation page + admin trail. The existing B1 anonymousâ†’user migration carries any saved trips with the new userId.

## Out of Slice D scope (deferred to D.x)

- **Real provider booking integrations** (`BookingComBookingProvider`, `ExpediaBookingProvider`, etc.). Same `BookingProvider` interface - drop-in replacement gated by per-provider keys + `STAYSCOUT_LIVE_BOOKING=1`.
- **Card capture / payment.** Even with live providers, Slice D's flow ends at the agreement to book. Real-mode integrations hand off to provider Checkout.
- **Autonomous mode.** `BookingAgent.draftBooking({ autonomous: true })` runs draft â†’ confirm in one shot. Premium gate via C4's `requirePremium(owner)`. Per-trip authorization scopes ("autonomous for this trip only").
- **LLM enrichment.** Free-text traveler parsing ("me + my partner + 2 kids"), date conflict resolution, special-request interpretation. Drafts in Slice D are deterministic transformations of saved-trip + form input.
- **Modifications.** Change dates, change guest count, change room type. D.x.
- **Group / multi-stay bookings.** Single hero stay only. Multi-leg itineraries become separate bookings in D.x.
- **`PostgresBookingStore` impl.** Schema lands in Slice D; impl when DB is available to integration-test.
- **Email confirmation.** No transactional email. The confirmation page is the only acknowledgment.
- **Refund settlement.** Mock cancellations are "fully refundable in dev." Real providers vary; D.x handles per-provider refund semantics.
- **Cron-based booking-status reconciliation.** A real provider booking can be modified out-of-band. A periodic reconciler that polls `provider.getBooking(id)` for non-terminal bookings lands in D.x.
- **Rate limiting** on `/api/bookings/confirm` beyond idempotencyKey coalescing.
