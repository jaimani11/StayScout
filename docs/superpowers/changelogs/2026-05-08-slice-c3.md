# Slice C3 — ItineraryAgent (multi-day plans)

**Tag:** `slice-c3`
**Plan:** [`docs/superpowers/plans/2026-05-08-slice-c3-itinerary.md`](../plans/2026-05-08-slice-c3-itinerary.md)

## Goal

Saved trips can be expanded into a day-by-day plan — activities, meals, transit, sequenced morning → late evening. The saved-trips panel grows a "Plan day-by-day" link; clicking opens a server-rendered page at `/trips/[tripId]/itinerary`. Mock-safe end-to-end via curated 3-day itineraries for the 7 Italian destinations + a generic synthesized fallback for everywhere else.

## What shipped

### Core types (`src/core/itinerary.ts`)
- `Itinerary { tripId, generatedAt, source, summary, days }`
- `ItineraryDay { dayNumber, theme, slots }`
- `ItinerarySlot { id, kind, startHint, title, detail, durationMinutes?, location?, costTier?, tags? }`
- Zod schemas for all of them; full validation across the boundary.

### Curated library (`src/lib/curation/itineraries.ts`)
- 7 hand-authored 3-day itineraries: Tuscany, Umbria, Amalfi, Rome, Venice, Lake Como, Cinque Terre.
- ~5 slots per day × 3 days × 7 destinations = ~105 slots of authored content.
- Voice rule enforced — every slot title + detail + day theme + summary passes `containsBannedWord`.
- Editorial tone matching the rest of the app (italic Fraunces fragments, no "discover/journey/unforgettable").

### Generators + Store + factory (`src/lib/itinerary/`)
- `ItineraryGenerator` interface — `generate(trip): Promise<Itinerary>`.
- `CuratedItineraryGenerator` — looks up by destination slug (matches via `findDestinationBySlugOrAlias`, so "Florence" → Tuscany template). Falls through to synthesized for non-curated destinations.
- `SynthesizedItineraryGenerator` — generic 3-day skeleton derived from `intent.vibe.tags` + `intent.travelers.groupKind`. Not great, never crashes.
- `InMemoryItineraryStore` — process-singleton cache (HMR-safe via globalThis); regeneration is cheap, cache is convenience.
- `getItinerarySubsystem()` — wires the lot.

### `/trips/[tripId]/itinerary` page
- Server-rendered (Next 16 server component). Owner-gated: resolves `getServerAuth()` + `getSessionStore().getTrip(...)`. 404s for trips not owned by the caller.
- Cache-first: subsystem store lookup; generates + caches on miss.
- `generateMetadata` returns hero + destination + nights for OG / browser title.
- `ItineraryView` renders the page with a Back-to-workspace link, hero header, three day blocks, and a closing italic note ("treat it as a frame, not a schedule").
- `SlotCard` chip renders kind + start hint + Fraunces title + italic detail + tags.

### Saved-trips integration
- Saved-trip row gains a "Plan day-by-day →" link in the bottom-right corner (next to the Bookmarked date).
- `data-row-action` keeps the click from bubbling to the row's resurface handler.
- No conflict with B3's Share/Remove buttons or B8's hover-action group — the new CTA is in the row footer, not the action cluster.

### Tests (+48)
- `tests/itinerary-curation.test.ts` (+24): every slug has a curated entry; each entry parses against `ItinerarySchema`; exactly 3 days; ≥4 slots/day; voice-lint pass on every text field; slot-id uniqueness within an itinerary.
- `tests/itinerary-generator.test.ts` (+8): curated path returns curated; alias resolution works ("Florence" → Tuscany); non-curated falls through to synthesized; vibe tags affect the synthesized copy; tripId is stamped per call.
- `tests/itinerary-store.test.ts` (+5): null on miss, round-trip, overwrite, invalidate, isolation by tripId.

## Architectural decisions (recorded)

**1. Itineraries are tied to saved trips.** No itinerary without `Save this trip` first. Reason: a 3-day plan is a richer commitment than a one-look proposal; it deserves a stable URL and persistence.

**2. The shape is editorial, not booking.** Slots are descriptive ("a slow walk down to Marina Grande"), not transactional. Reason: a 3-day plan you can adjust beats a rigid schedule the model will get wrong on dates. C3.x layers Viator activity search to add bookable items to specific slots.

**3. Generator behind one interface.** `CuratedItineraryGenerator` for known destinations, `SynthesizedItineraryGenerator` for the rest, both returning the same `Itinerary` shape. `ModelItineraryGenerator` (live model) drops in via the same interface in C3.x.

**4. Curated coverage is real, not stub.** All 7 Italian destinations have full hand-authored 3-day itineraries. A demo against Tuscany / Rome / Venice etc. shows real planning quality.

**5. Generic synthesized fallback never crashes.** Any destination outside the curated library still produces a valid 3-day frame derived from intent.

**6. Cache-first, regeneration cheap.** Curated lookup is O(1); the cache is convenience, not correctness. Postgres-persisted itineraries (with editing) land in C3.x.

**7. Owner-gated at the page.** The route 404s when the requester doesn't own the underlying SavedTrip. No new auth surface — same key as everything else (B1).

## Demo behavior

Save a trip ("Tuscany, slow and walkable" → Save). Open the saved-trips panel. The row's footer shows a "PLAN DAY-BY-DAY →" link. Click it: opens `/trips/[tripId]/itinerary` with three days of curated Tuscany content (Florence morning + Boboli, Chianti wine country, Siena + San Gimignano). Refresh: the cache hits, no regeneration.

## Out of C3 scope (deferred to C3.x)

- `ModelItineraryGenerator` — live model + activity-provider context. Lands when Viator is wired.
- Editing slots (drag-drop, manual swaps).
- Sharing an itinerary (separate share-slug surface).
- Stripe gate (C4 will lock the CTA behind premium for non-curated destinations).
- PDF / iCal export.
- Postgres-persisted itineraries.
