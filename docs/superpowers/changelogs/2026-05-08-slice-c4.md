# Slice C4 - Stripe (premium tier)

**Tag:** `slice-c4`
**Plan:** [`docs/superpowers/plans/2026-05-08-slice-c4-stripe.md`](../plans/2026-05-08-slice-c4-stripe.md)
**Setup guide:** [`docs/billing.md`](../../billing.md)

## Goal

Open a real paid tier with Stripe Checkout. The "Plan day-by-day" CTA in the workspace gets a soft gate: when the destination falls through to the synthesized itinerary path AND the caller's entitlement is free, the page renders an upgrade card instead. Curated Italy stays free for everyone - the demo never breaks. Mock-safe end-to-end without keys; with `STRIPE_SECRET_KEY` + `STRIPE_WEBHOOK_SECRET` + `STRIPE_PRICE_ID` set, real Checkout + webhook flips the subscription on.

C4 ships **both** providers fully implemented + tested. The mock isn't a stub; it's the keyless-dev default and the lib-test substrate. Real Stripe integration is the headline change - fixture-signed integration tests prove the verification + idempotency + state-sync code paths run end-to-end.

## What shipped

### Core types (`src/core/billing.ts`)
- `Plan = 'free' | 'premium'`.
- `EntitlementSource` - telemetry/UI flag for *why* the entitlement is what it is (`mock-everyone-premium`, `mock-anonymous`, `stripe-active`, `stripe-trialing`, `stripe-canceled-grace`, `free`).
- `Entitlement { plan, premiumUntil, source }`.
- `Subscription { ownerKind, ownerId, stripeCustomerId, stripeSubscriptionId, priceId, status, currentPeriodEnd, updatedAt }` - `status` mirrors Stripe's enum exactly so webhook payloads store verbatim.
- `OwnerKey` re-defined here (structurally compatible with `ownerOf(authState)`) so the core layer stays free of `@lib/auth` dependencies.
- Zod schemas for everything; validation across the boundary.
- `isPremiumStatus(status, currentPeriodEnd)` - derives premium-or-not (canceled-but-still-in-grace counts as premium).
- `entitlementFromSubscription(sub)` - read-model derivation; one place owns the rule.
- `BillingError` - typed checkout failures (`sign-in-required`, `misconfigured`, `stripe-error`, `no-checkout-url`).

### `BillingProvider` interface + two impls (`src/lib/billing/`)

**Interface** (`billing-provider.ts`):
```ts
interface BillingProvider {
  kind: 'mock' | 'stripe';
  getEntitlement(owner): Promise<Entitlement>;
  createCheckoutSession({ owner, returnUrl, cancelUrl }): Promise<{ url }>;
  handleWebhook({ rawBody, signature }): Promise<HandleWebhookResult>; // never throws
}
```

**`MockBillingProvider`** (`mock-billing-provider.ts`):
- Anonymous owners → free (`source: 'mock-anonymous'`).
- Authenticated owners → premium (`source: 'mock-everyone-premium'`).
- `createCheckoutSession` → URL into the in-app `/billing/mock-checkout` page (a redirect-cycle stand-in for Stripe Checkout).
- `handleWebhook` → no-op; the route returns 503 with a warning so a mis-pointed Stripe Dashboard fails loudly.

**`StripeBillingProvider`** (`stripe-billing-provider.ts`):
- Wraps the official `stripe` SDK (v22, API version `2026-04-22.dahlia` pinned).
- `getEntitlement` reads our `SubscriptionStore`. Anonymous → free. User with no row → free. Otherwise derived from `entitlementFromSubscription`.
- `createCheckoutSession`:
  - Anonymous → throws `BillingError('sign-in-required')`.
  - User → lookup-or-create customer (eagerly persists `stripeCustomerId` so retries don't dupe). Creates Checkout session in `mode: 'subscription'` with `client_reference_id` + `metadata.{ownerKind,ownerId}` on both the session and `subscription_data` so the webhook can attribute back to the owner.
  - Returns the `session.url`.
- `handleWebhook`:
  - No signature header → `{ ok: false, reason: 'no-signature' }`.
  - `stripe.webhooks.constructEvent(rawBody, signature, secret)` for verification. Failure → `{ ok: false, reason: 'signature' }`. Logged, never thrown.
  - `WebhookEventStore.markProcessed(event.id)` atomic check-and-set. Duplicate → `{ ok: true, idempotent: true }` (do NOT re-apply).
  - `checkout.session.completed` → fetch the subscription via `stripe.subscriptions.retrieve(...)`, upsert with priceId/status/currentPeriodEnd.
  - `customer.subscription.created` | `customer.subscription.updated` → upsert from the event payload directly.
  - `customer.subscription.deleted` → flip status to 'canceled' via reverse-lookup by `stripeSubscriptionId`, preserve `currentPeriodEnd` for grace.
  - Other types → `{ ok: true, ignored: true }`. Apply errors → logged + 200 (Stripe's retry won't help; an admin can replay from the Dashboard).
- Stripe SDK API moved `current_period_end` from `subscription` root onto `subscription.items.data[].current_period_end` in the dahlia release; we collapse via a single `readCurrentPeriodEnd()` helper for storage.

### Stores (`src/lib/billing/`)
- `SubscriptionStore` interface - `getByOwner`, `upsert`, `setStatusByStripeSubscriptionId` (reverse-lookup for webhooks that arrive carrying only the Stripe id).
- `InMemorySubscriptionStore` - process-singleton via globalThis, two indexes (byOwner + byStripeSubscriptionId) kept in sync, dedupe defense for changed `stripeSubscriptionId`.
- `WebhookEventStore` interface - atomic `markProcessed(id) → 'new' | 'duplicate'`.
- `InMemoryWebhookEventStore` - bounded LRU (default 1000), oldest-first eviction.
- Postgres impls deferred to C4.x (schema is in place - see below).

### Server-side gate (`src/lib/billing/gates.ts`)
- `requirePremium(owner)` - single seam every UI/route consumer uses.
- Returns `{ entitled: true, entitlement }` or `{ entitled: false, entitlement, reason: 'free' | 'anonymous' }`.
- Returns the entitlement either way so upgrade-card UI knows what reason to render.

### Factory (`src/lib/billing/factory.ts`)
- `getBillingSubsystem(): { provider, store, eventLog, kind }`.
- All three Stripe vars set → `StripeBillingProvider`.
- Any subset present (partial config) → `MockBillingProvider` + `console.warn` naming the missing vars (so dev never silently breaks).
- HMR-safe singleton.

### API routes (`src/app/api/billing/`)
- `POST /api/billing/checkout` - owner-gated, body `{ returnUrl, cancelUrl }` (Zod-parsed), 401 for anonymous (with `error: 'sign-in-required'`), `BillingError` mapped to 400/401, returns `{ url }`.
- `POST /api/billing/webhook` - `runtime: 'nodejs'`, `dynamic: 'force-dynamic'`. Reads raw body via `req.text()` (signature verification needs the bytes verbatim). Maps results: signature failure → 400, mock-mode delivery → 503 with warning, success → 200 with `{ ok, eventType, eventId, idempotent?, ignored? }`. Never throws - Stripe's retry on 5xx would be useless if the issue is on our side.
- `GET /api/billing/entitlement` - owner-aware read. Used by the post-checkout return page to poll until the webhook lands.

### UI (`src/features/billing/` + `src/app/billing/`)
- `UpgradeCard` (server component): Soft paywall on the itinerary page. Heading "Day-by-day for everywhere - premium." + voice-matching italic body + bulleted feature list + reason-aware CTA. Renders for anonymous (CTA: sign-in-then-upgrade) and free authed (CTA: direct upgrade).
- `UpgradeCta` (client component): the actual button. POSTs `/api/billing/checkout` with `{ returnUrl, cancelUrl }` + redirects to the returned URL. For `signInRequired` cases, redirects to `/sign-in?redirect_url=...` instead.
- `EntitlementBadge`: tiny "PREMIUM" geist-mono chip. Renders nothing for free/anonymous so server consumers can drop it in unconditionally.
- `/billing/return` (server) + `poll-entitlement.tsx` (client): the post-Checkout landing. Polls `/api/billing/entitlement` every 1s up to 15s; on premium → redirect to the original itinerary; on timeout → soft message + manual link.
- `/billing/mock-checkout` (server): Stand-in for Stripe Checkout when in mock mode. Two server actions ("Pretend to pay" + "Cancel") that just redirect - MockBillingProvider already grants premium to authed users; this page exists so the flow shape is identical to real Stripe and the demo is exercisable from the UI.

## Architectural decisions (recorded)

**1. Stripe-first, mock-fallback.** C4 ships `StripeBillingProvider` as a fully-tested first-class implementation, not a "land later" stub. The mock provider remains as the keyless-dev default and the lib-test substrate. Reason: the user explicitly wanted real Stripe-ready architecture in C4, and the test infrastructure (SDK fixture-signing) is good enough to prove the verification + idempotency + state-sync code paths without live network.

**2. Webhook idempotency on `event.id`, atomic check-and-set.** Stripe retries deliveries on any non-2xx and occasionally double-delivers. `WebhookEventStore.markProcessed` returns `'new'` exactly once per id; second call returns `'duplicate'` and the handler short-circuits. Concurrent calls for the same id resolve with exactly one `'new'`. Covered in tests.

**3. Subscription state is server-side, every render.** `requirePremium(owner)` always reads `provider.getEntitlement(owner)`. No client-claimed premium flag. Even after Checkout success, the return page polls - it never trusts a query param. The only mutation path to premium state is a verified webhook delivery.

**4. Raw body for signature verification.** The webhook route reads `req.text()` directly. Next 16's default body parsing doesn't run on this route (`runtime: 'nodejs'`, `dynamic: 'force-dynamic'`). The bytes Stripe signed are the bytes we verify.

**5. Anonymous can't checkout.** `createCheckoutSession` throws `BillingError('sign-in-required')` for `ownerKind === 'session'`. The route maps that to 401. The upgrade card sends anon users to `/sign-in?redirect_url=...` so they bounce back; the existing B1 anonymous→user migration carries any saved trips/memories with the new userId.

**6. Refund + cancellation grace.** `customer.subscription.deleted` flips status to `'canceled'` but preserves `currentPeriodEnd`. `entitlementFromSubscription` returns premium with `source: 'stripe-canceled-grace'` while we're still inside that period. Honors what the user paid for.

**7. PCI scope = zero.** Stripe Checkout (hosted page) handles cards. We store: `stripeCustomerId`, `stripeSubscriptionId`, `priceId`, `status`, `currentPeriodEnd`. That's the entire surface.

**8. One feature gated in C4.** Synthesized-itinerary fallback is the only premium gate. Curated Italian destinations bypass it (the demo always works). Future gates layer on the same `requirePremium(owner)` seam without touching the billing lib.

**9. Partial config fails loud.** If `STRIPE_SECRET_KEY` is set but `_WEBHOOK_SECRET` or `_PRICE_ID` is missing, the factory logs a clear warning naming the missing var and falls back to mock. Dev never silently misroutes.

**10. Production switch is no-code.** Same `StripeBillingProvider` runs against `sk_test_...` (development) or `sk_live_...` (production). Just swap env vars + configure a permanent webhook in the Stripe Dashboard.

## Mock-safe matrix

| Vars | Provider | Behavior |
|---|---|---|

| All three: `STRIPE_SECRET_KEY` + `STRIPE_WEBHOOK_SECRET` + `STRIPE_PRICE_ID` (test or live) | Stripe | Real Checkout + webhook. Subscription state is source of truth. |
| Partial: any subset of the three | Mock + warning | Logs which var(s) are missing, falls back to mock. |

## Demo behavior

**Mock mode (no keys)**:

1. Sign in.
2. Save a Tokyo trip.
3. Click "PLAN DAY-BY-DAY →" (lock glyph visible - non-curated destination).
4. Upgrade card renders.
5. Click "Upgrade to Premium" → redirects to `/billing/mock-checkout`.
6. Click "Pretend to pay" → bounces to `/billing/return` → polls `/api/billing/entitlement` (which returns premium because mock provider grants it for any authed user) → redirects to the itinerary URL.
7. Synthesized Tokyo itinerary renders. Curated Tuscany continues to render with no gate.

**Stripe test-mode** (full walkthrough in `docs/billing.md`):

1. `stripe listen --forward-to localhost:3000/api/billing/webhook` → copy `whsec_...`.
2. Set `STRIPE_SECRET_KEY` (`sk_test_...`), `STRIPE_WEBHOOK_SECRET`, `STRIPE_PRICE_ID` in `.env.local`.
3. Restart `pnpm dev`.
4. Same flow - but step 5 redirects to real Stripe Checkout.
5. Pay with `4242 4242 4242 4242` (any future expiry, any 3-digit CVC, any zip).
6. Stripe redirects to `/billing/return`. The CLI logs `--> checkout.session.completed [evt_...]` + `<-- [200] POST /api/billing/webhook`. Within ~1s the poll picks up `plan === 'premium'` and bounces to the itinerary URL.
7. Verify in Stripe Dashboard → Customers (the customer + active subscription are there).
8. `stripe trigger customer.subscription.updated` / `.deleted` to test status changes - webhook applies them, `getEntitlement` reflects the new state immediately.
9. Re-deliver an event from Dashboard → returns 200 with `idempotent: true`, store state unchanged.

## Out of C4 scope (deferred to C4.x)

- `PostgresSubscriptionStore` impl (schema lands in C4; impl in C4.x with a deployed DB to integration-test against - `INSERT ON CONFLICT DO NOTHING` for true atomic event-id dedupe).
- Refunds + proration handling beyond what Stripe surfaces in `customer.subscription.updated`.
- Annual vs monthly + multi-tier (C4 ships single "Premium" plan).
- Stripe Customer Portal (cancel + update card without leaving the app).
- Per-feature granular gating beyond synthesized-itinerary (monitoring on >3 trips, advanced share/embed).
- Email/transactional notifications on subscription change.
- Live integration tests against a real Stripe test account in CI (current C4 tests use SDK fixture-signing, which exercises the same verification code path without network).
- A cron/job that auto-flips entitlement off when `currentPeriodEnd` passes (today the next `getEntitlement` derivation handles it correctly, but stale `Subscription.status='canceled'` rows accumulate).
- `ModelItineraryGenerator` itself - C3 deferred this; C4 unlocks the demand for it but the impl arrives when Viator is wired.
