# Slice C5 — Admin panel extensions

**Tag:** `slice-c5`
**Plan:** [`docs/superpowers/plans/2026-05-08-slice-c5-admin.md`](../plans/2026-05-08-slice-c5-admin.md)

## Goal

Grow `/admin` from a telemetry summary into a real operator console. Four new server-rendered pages — per-turn drill-in, affiliate click feed, per-owner aggregate, memory index browser — plus a billing summary card on the dashboard. All views read through existing subsystem interfaces; the only new code in `lib/` is small, additive enumeration helpers (`listClicks`, `listForOwner`, `listAllOwners`) and a single shared admin auth gate.

## What shipped

### `requireAdmin()` (`src/lib/admin/require-admin.ts`)
- Single decision point for admin auth across every admin page.
- Three modes: Clerk-on requires sign-in (else `redirect('/')`); Clerk-off open (keyless dev); `STAYSCOUT_ADMIN_PUBLIC=1` open in any mode (staging).
- Returns the resolved `AuthState` so callers that want to label rows ("you, viewing as <email>") can do so without re-resolving.
- Replaces the inline gate that previously lived only in `/admin/page.tsx`.

### Shared admin shell (`src/features/admin/`)
- `AdminShell` — wraps a page with the breadcrumb header + nav. Same width and padding as the dashboard so navigation between pages feels stable.
- `AdminNav` — top horizontal nav (Dashboard / Turns / Clicks / Users / Memories) with active-link accent border.
- `DataTable<T>` — bare table primitive with `columns: { key, label, render, width? }` + empty-state. No deps, no virtualization (lists are bounded ≤100).
- `OwnerLink` — `<owner kind chip + truncated id>` linking to `/admin/users/[ownerId]?kind=<kind>`. Used everywhere an owner reference appears.

### `/admin/turns/[turnId]` (`src/app/admin/turns/[turnId]/page.tsx`)
- Drill-into-a-trace.
- Header card: turn id, status (color-coded), type, duration, total cost, total tokens, started-at, session (linked to `/admin/users/<sessionId>?kind=session`).
- Failure block when `status === 'failed'`: `failureError` rendered in a warning-bordered card.
- Agent timeline `<DataTable>` columns: agent, duration, model, tokens (in→out), cost (only when any agent reported it), cache (only when relevant), error.
- 404 when the turn has scrolled out of the in-process ring buffer.
- Reads via the new `MemoryTelemetryStore.getTurn(turnId)` accessor.

### `/admin/clicks` (`src/app/admin/clicks/page.tsx`)
- Affiliate click feed, most-recent-first.
- Header subtitle: `<count> recent · <conversions> converted`.
- Columns: time, owner (OwnerLink), provider, stay tail, turn (links to `/admin/turns/[turnId]` when present).
- Empty state: italic "No clicks recorded yet — go save a trip and tap a hero card."

### `/admin/users/[userId]` (`src/app/admin/users/[userId]/page.tsx`)
- Per-owner aggregate. The `[userId]` segment is the literal id; `?kind=user|session` disambiguates anonymous vs authenticated owners.
- Header card: kind chip + id + four stat values (saved trips, clicks, memories, plan), with entitlement source + `premiumUntil` line.
- Four sections (each with a 10-row preview + "all memories →" link if applicable):
  - Saved trips (destination, hero stay, nights, bookmarked).
  - Affiliate clicks (time, provider, stay tail, turn link).
  - Memories (kind chip, italic content, when).
  - Recent turns from this session (only for `ownerKind === 'session'`, since the telemetry store is keyed by sessionId).

### `/admin/memories` (`src/app/admin/memories/page.tsx`)
- Two modes via search params:
  - No `?owner=`: list all owners with memory counts, ordered by latest activity. Click → drill into per-owner view.
  - `?owner=<kind>:<id>`: list that owner's memories most-recent-first. With `?q=<query>` also set, runs `MemoryStore.search` and shows ranked results with cosine scores in a leading column.
- The form posts back to `/admin/memories?owner=<...>&q=<...>` so query params drive everything; no client state.
- Operator's window into the in-memory ring buffer. Postgres path follows when C1.x lands its store impl.

### Dashboard updates (`src/app/admin/page.tsx`)
- Replaced inline auth gate with `requireAdmin()`.
- Wrapped in `<AdminShell section="dashboard">` so it shares the new nav.
- Added a fifth summary card: `Billing` — shows `mock` or `Stripe` (emphasized when stripe), with the right caption underneath.
- `TurnRow` rows now link the turn id to `/admin/turns/[turnId]` and the session id to `/admin/users/[sessionId]?kind=session`. Drill-in is one click from the dashboard.

### Store extensions (only-additive)
- `SessionStore.listClicks({ owner?, limit? })` — admin needs an enumeration; route handlers never use it. In-memory + Postgres impls both ship in C5.
- `MemoryStore.listForOwner({ owner, kind?, limit? })` — most-recent-first list. Most useful for the per-owner page + the memories table when `?q=` is absent.
- `MemoryStore.listAllOwners()` — distinct owners with at least one record. In-memory only in C5; Postgres impl lands when C1.x does.
- `MemoryTelemetryStore.getTurn(turnId)` — public accessor over the existing private `turnIndex` map.

### Tests (+13, total now 418)
- `tests/admin-store-extensions.test.ts` (+9): listClicks order/limit/owner-filter, listForOwner order, kind-filter, unknown-owner empty, listAllOwners distinct, listAllOwners empty.
- `tests/admin-require-admin.test.ts` (+4): keyless dev path, public override path, gated path (redirect throws sentinel), authenticated pass-through. `next/navigation` is mocked to throw a sentinel `__redirect__:<url>` so the gated case is testable.

## Architectural decisions (recorded)

**1. Surface, don't query.** Every admin page reads through existing interfaces (`SessionStore`, `MemoryStore`, `BillingSubsystem`, `MemoryTelemetryStore`). When data was already there but not enumerable (clicks list, memory list-by-owner, single turn lookup), we added a method on the interface — both impls implement it. No raw Prisma queries from pages.

**2. Owner-keyed everywhere.** The aggregate route at `/admin/users/[userId]` is "owner-keyed" in spirit — anonymous sessions have data too. `OwnerLink` and the `?kind=` query param disambiguate; both kinds get the same affordance. URL stays `[userId]` for path readability.

**3. One auth gate, three modes.** `requireAdmin()` is the single decision point. Every admin page calls it as the first line. The behavior matrix (Clerk on/off × public override) is documented inside the helper itself.

**4. Pages are server components.** Lists render with the data already in hand; no client-side data-fetching, no client-side state for the filter form (the search form posts to itself). The only client interactivity in C5 is the existing dashboard chart from B7.

**5. Mock-safe + Postgres-ready.** All three new interface methods have an in-memory impl from day one. The Postgres `listClicks` ships in C5; the Postgres MemoryStore methods queue behind C1.x's store impl.

**6. Voice + design vocabulary unchanged.** Same Fraunces / Inter / geist-mono palette, surface hierarchy, and breadcrumb pattern as the dashboard. New pages feel like extensions of the dashboard, not a separate area.

**7. No new auth surface.** All admin pages share the single gate. No per-page permission system, no `/admin/users/[userId]` "edit" affordance — read-only operator console in C5.

**8. Telemetry data is intentionally ephemeral.** Drill-in returns `notFound()` when the turn has scrolled out of the ring buffer. Operators get the local trace + a Langfuse link (when wired) for the durable copy.

## Demo behavior

1. Open `/admin` (in mock mode, no Clerk env, you go straight in). Five summary cards across the top — note the new **Billing** card showing `Mock`.
2. Click any recent turn id → `/admin/turns/[turnId]`. Header shows status + duration + total cost; the agent timeline lists each step (intent, mood, planner, etc.) with its tokens + duration.
3. From the turn detail, click the session link → `/admin/users/<sessionId>?kind=session`. See trips saved, clicks made, memories recorded, plan (Free in mock for anon), entitlement source.
4. Click the **Memories** nav link → list of owners with counts. Click "View →" → owner's memory list. Type a query, hit Search — same page renders with similarity scores.
5. Click **Clicks** nav → affiliate-click feed. Each row's session/user is linked back to its owner page; each turn id is linked back to its trace.

## Out of C5 scope (deferred)

- Postgres `MemoryStore.listForOwner` + `listAllOwners` — wired when C1.x lands its impl.
- Search/filter UI on clicks (date range, provider filter) — current view is most-recent-first only.
- Aggregated revenue / conversion-rate analytics on clicks.
- Audit log of admin actions (read-only in C5 anyway).
- Per-page authz beyond the single gate.
- Real-time updates (no SSE on admin pages — refresh to see new data).
- Edit affordances (delete trip, clear memories, etc.).
