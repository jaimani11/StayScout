# Slice C2 - MonitoringAgent (saved-trip watcher)

**Tag:** `slice-c2`
**Plan:** [`docs/superpowers/plans/2026-05-08-slice-c2-monitoring.md`](../plans/2026-05-08-slice-c2-monitoring.md)

## Goal

Saved trips become living. When the price drops, availability changes, or a featured stay opens up, the saved-trips panel surfaces a small badge on the row ("↓ 12% since saved"). Mock-safe end-to-end - the dev demo produces visible, deterministic monitoring events without real providers or DB.

## What shipped

### Monitoring primitives (`src/lib/monitoring/`)
- `MonitoringEvent`, `MonitoringEventKind`, `MonitoringSnapshot` types - append-only event log + per-trip snapshot.
- `MonitoringStore` interface + `InMemoryMonitoringStore` impl - process-singleton (HMR-safe via `globalThis`), maps for snapshots + per-owner event lists.
- `MonitoringChecker` interface + `MockMonitoringChecker` - deterministic synthetic events seeded on `(tripId, minute-of-now)`. Weighted lottery: 45% price-drop, 15% price-rise, 10% better-match, 5% new-alternative, 5% unavailable, 20% no-event.
- `MonitoringRunner.checkOwner({owner, trips, now})` - on-demand entry point. Throttles per-trip checks at 60s (env-overridable via `STAYSCOUT_MONITORING_INTERVAL_MS`); runs the checker; persists snapshots + events; returns the full unacknowledged-event list grouped by trip.
- `getMonitoringSubsystem()` - process singleton; cached `{store, checker, runner, kind}`.

### API surface
- `GET /api/trips/list` - enriched response: each trip now has `monitoringEvents: MonitoringEvent[]`. Monitoring runs are wrapped in try/catch so a runner failure never blocks the trips response.
- `POST /api/trips/[tripId]/monitoring/acknowledge` - owner-gated. Marks every unack event for the trip as read. Idempotent. Defense-in-depth: confirms the caller owns the trip before flipping events.

### UI
- `MonitoringBadge` - chip on the saved-trip row. Color-coded by kind (price-drop = green, price-rise/unavailable = warning amber, others = neutral). Renders the most recent unack event (single-badge UX). Hover surfaces the full message; `aria-label` for screen readers.
- `useSavedTrips` - `monitoringEvents` on `SavedTripRow`, plus `acknowledgeMonitoring(tripId)` (optimistic local clear + background POST).
- `SavedTripRow` - renders the badge between the destination line and the hero name when events exist.
- `SavedTripsPanel.handleSelect` - clicking a row to resurface also fires `acknowledgeMonitoring(trip.id)` so the badge clears.

## Architectural decisions (recorded)

**1. Pull, not push.** No WebSockets, no SSE, no background scheduler. The runner triggers on `/api/trips/list` calls. Reason: serverless-friendly, demo works without process-level singletons, and "I refreshed and it has updates" is the natural mental model.

**2. The checker is real OR mock, behind one interface.** C2 ships only the mock (synthetic events seeded for stability). Real provider re-checking lands in C2.x.

**3. Trip-scoped throttling.** Each trip is checked at most every `STAYSCOUT_MONITORING_INTERVAL_MS` (default 60s). Newly-saved trips: snapshot at save time; first event possible on the next list-fetch ≥ 60s later (avoids confusing immediate-event UX).

**4. Event budget = 0 or 1 per check.** Single visible badge per row keeps the UI calm.

**5. Owner-scoped, append-only.** Events never delete; `acknowledged` flips on user interaction. Audit trail for B-late admin.

**6. Mock-safe is the invariant.** Without DB / real provider keys, the mock checker produces synthetic events; the in-memory store persists across HMR. Adding real keys (C2.x) swaps the checker without touching the runner or UI.

**7. Failure isolation everywhere.** Per-trip try/catch in the runner; route-handler try/catch for monitoring; optimistic UI with background POST for acknowledge. Monitoring NEVER blocks the user-visible flow.

## Demo behavior

After C2: save a trip ("Tuscany, slow and walkable" → Save this trip). Wait 60+ seconds. Open the saved-trips panel. The row now shows a small chip - usually a green "↓ 5%" or similar. Click the row: resurfaces as before, badge clears.

## Out of C2 scope (deferred to C2.x)

- `RealMonitoringChecker` impl - re-runs `provider.search` + diffs against saved proposal. Needs real provider keys + cache invalidation tuning.
- `PostgresMonitoringStore` - schema + migration. Drop-in once DB is wired.
- Background scheduler (Vercel cron / `pg_cron`) - push model. Pull works for the demo + is forward-compatible.
- Email / push notifications.
- Cross-trip "you might like this new destination" suggestions - Slice D territory.
